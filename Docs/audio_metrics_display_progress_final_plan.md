# 音声実測値・表示・進行状況の最終修正方針

## 結論

今回の修正は、次の3系統をまとめて扱う。

| 系統 | 修正目的 |
|---|---|
| ノイズチェック表示 | 実測値はそのまま使い、バー表示の誤解をなくす |
| 補正・マスタリング進行状況 | ログ文字列依存ではなく、工程ごとに正しく進む表示へ直す |
| LUFS / True Peak / LRA | 測定基準を一本化し、表・グラフ・品質チェックで同じ値を使う |

これは見た目だけの調整ではない。
測定値、表示用変換、工程状態、音量系測定の責務を分ける修正である。

最終方針は次の一文に集約する。

> 実測値は変えず、表示は誤解なく、進行状況は工程単位で、音量系指標は同じ測定基準で出す。

## 1. ノイズチェック表示の修正方針

### 現状の事実

| 項目 | 事実 |
|---|---|
| 入力ノイズ値 | `NoiseMeasurementService.analyze(signal:)` で実音声から測定している |
| 表示値 | `NoiseCheckReportService` が `measuredLevelDB` を画面表示へ渡している |
| 問題 | 数値そのものではなく、バー長へ変換する表示ロジックが誤解を生む |
| 現在のバー | 固定レンジに対する絶対値バー |
| 問題点 | 実測差よりも、補正後バーが極端に短く見える |
| 未測定バー | `--` の行もバー領域に出て、不自然に見える |
| 判断への影響 | ユーザーが「実測値以上に大きく変化した」と誤認しやすい |

### 修正の原則

| 対象 | 方針 |
|---|---|
| 実測値 | 変更しない |
| 差分値 | 変更しない |
| 判定文 | 実測値から出す |
| バーの長さ | 表示用の見え方として調整する |
| 3本構成 | 入力 / 補正後 / 最終版 を維持 |
| 未測定 | 同じ長さの薄いグレーで統一し、未実行と分かる表示にする |

音声分析の数値はそのまま信じる。
ただし、バーを数値そのものの絶対値として見せすぎないようにする。

## 2. ノイズバー表示の具体仕様

### 現在避けたい表示

```text
入力      █████████████████████
補正後    ██
最終版    ─
```

この表示は、数値差よりも大きく差があるように見える。

### 修正後の考え方

```text
入力      ███████████
補正後    ████████
最終版    ─────────
```

| 表示 | 意味 |
|---|---|
| 青バー | 入力音源の実測値 |
| 緑バー | 補正後の実測値 |
| オレンジバー | 最終版の実測値 |
| 薄いグレー | 未測定、または未実行 |
| 数値 | 実測値そのもの |
| 差分 | 入力比、または補正後比の変化量 |
| 判定 | 実測差から出した文章 |

### バーの幅

| 項目 | 仕様 |
|---|---|
| バー領域 | カード全幅ではなく、およそ 2/3 程度 |
| 数値領域 | 右側に確保 |
| 判定領域 | 右上、または右側に読みやすく配置 |
| 文字サイズ | 現状より小さくしすぎない |
| 未測定バー | 全ノイズ種別で同じ長さ、同じ薄さ |

バーは正確なメーターではなく、変化を直感的に見る補助とする。
正確な値は、右側の数値と差分に出す。

## 3. ノイズ種別ごとの改善方向

ノイズは、すべて同じ意味ではない。
そのため、各項目に短い説明を付ける。

| ノイズ | 単位 | 改善方向 | 表示説明 |
|---|---:|---|---|
| ヒス・シュワシュワ | dBFS | 下がるほど良い | 下がるほどノイズが少ない |
| 高域のチラつき | dBFS | 下がるほど良い | 下がるほど高域ノイズが少ない |
| 低域ゴロゴロ | dBFS | 下がるほど良い | 下がるほど低域ノイズが少ない |
| 環境音・部屋鳴り | dBFS | 下がるほど良い | 下がるほど環境音が少ない |
| ハム・電源ノイズ | dB | 下がるほど良い | 下がるほど電源ノイズが少ない |
| サ行・歯擦音 | dB | 増えないことが重要 | 増えると刺さりやすい。下げすぎると声が丸くなる |
| こもり・低いザラつき | dB | 上がると悪化 | 上がるとこもりやすい |

`サ行` は単純に「下がれば良い」とは言い切れない。
下げすぎると、声の明瞭さが落ちる可能性がある。

## 4. ノイズチェックの判定仕様

判定は、バーの長さではなく実測値から出す。

| 判定 | 条件の考え方 |
|---|---|
| 大きく改善 | 原音より明確にノイズが下がった |
| 改善 | 原音より下がった |
| ほぼ維持 | 差が小さい |
| 戻りあり | 補正後より最終版で増えたが、原音よりは低い |
| 原音より悪化 | 原音より悪い方向へ増えた |
| 悪化なし | サ行などで増加が許容範囲内 |

### 表示例

```text
ヒス・シュワシュワ
下がるほどノイズが少ない

入力      ███████████        -101.8 dBFS
補正後    ████████           -116.7 dBFS   -14.9 dB 大きく改善
最終版    ████████           -115.9 dBFS   +0.8 dB 少し戻り

判定: 原音より大幅に低い / 少し戻りあり
```

バー、数値、差分、判定の意味を分ける。

## 5. 進行状況表示の修正方針

### 現状の問題

| 現在 | 問題 |
|---|---|
| ログ文字列に反応して進む | ログ文が変わると進行表示が壊れやすい |
| 長い工程中に止まって見える | 実際は処理中でもユーザーには停止に見える |
| スキップが推測扱い | なぜ灰色なのか分かりにくい |
| 工程完了のタイミングが曖昧 | 次の工程に入って初めて前工程が終わったように見える |

### 修正後の状態

| 状態 | 表示 |
|---|---|
| 未開始 | 薄いグレー |
| 実行中 | 現在工程として表示 |
| 完了 | 緑チェック |
| スキップ | グレーのスキップ表示 |
| 失敗 | 赤、または警告表示 |

進行状況はログ文から推測しない。
処理側から、工程イベントを明示的に出す。

## 8. LUFS / True Peak / LRA の修正方針

ここはノイズバーとは別問題である。
目的は、音量系の数値を1つの基準にそろえること。

### 現状の問題

| 項目 | 問題 |
|---|---|
| LUFS | アプリ内の目安値として近い計算をしているが、表示上は正式値に見えやすい |
| True Peak | `approximateTruePeak` で近似だが、画面では正式値のように見える |
| LRA | 簡易的なラウドネス範囲で、厳密なLRAとは言い切れない |
| 使用箇所 | 表、グラフ、品質チェックで計算入口が分散しやすい |

### 修正方針

`LoudnessMeasurementService` を作り、以下をまとめる。

| 指標 | 測定方針 |
|---|---|
| LUFS | K-weighting、400msブロック、100ms hop、絶対ゲート、相対ゲート |
| True Peak | 4倍以上のオーバーサンプリングで測定方法を固定 |
| LRA | LUFSと同じ短期ラウドネスを元に、下位10%〜上位95%の差で算出 |

### 表示方針

| 場合 | 表示 |
|---|---|
| 正式測定寄りに整える | `音量 -18.2 LUFS` |
| 近似のまま残す | `音量感 -18.2 LUFS目安` |
| True Peak を近似のまま残す | `ピーク目安` または `Approx. True Peak` |
| 測定を固定できた場合 | `True Peak` として表示 |

今回は、音質判断に使うため、測定基準を固定して正式測定寄りに整える。

## 9. 責務分離の設計

現在の問題は、計算と表示が近くなりすぎている点である。
そこで役割を分ける。

| 役割 | 担当 |
|---|---|
| 音声処理 | 補正・マスタリングそのもの |
| ノイズ測定 | `NoiseMeasurementService` |
| ノイズ表示用レポート | `NoiseCheckReportService` |
| バー表示スケール | `NoiseCheckDisplayScale` |
| 音量系測定 | `LoudnessMeasurementService` |
| 比較用データ作成 | `AudioComparisonService` |
| マスタリング判断 | `MasteringAnalysisService` |
| 画面表示 | `ContentView` |

この分け方にすると、次を防げる。

- 実測値を表示都合で変えてしまう
- 表とグラフで違う値が出る
- ログ文変更で進行状況が壊れる
- 近似値なのに正式値のように見える

## 10. 実装計画

### Step 1: ノイズ表示スケールの調査と固定

- `NoiseCheckDisplayScale` の現在の変換方法を確認する
- 各ノイズ種別の単位と改善方向を整理する
- 実測値と表示用バー長を分ける

### Step 2: ノイズバーの表示用変換を修正

- 実測値はそのまま保持する
- バーだけ、差が誇張されすぎないよう変換する
- 未測定バーを全行で統一する
- バー領域をカード幅の約2/3に制限する

### Step 3: ノイズ説明をモデルへ追加

- `NoiseCheckRow` または相当モデルに説明文を追加する
- `下がるほどノイズが少ない` などを画面に出す
- サ行とこもりは単純な上下判定にしない

### Step 4: ノイズチェックUIを調整

- 数値を読みやすくする
- 差分表示を読みやすくする
- 判定文を見やすい位置へ置く
- 3本バー構成は維持する

### Step 5: ノイズバーのテスト追加

- `-101.8 → -116.7` のような差で補正後バーが極端に短くならないことを確認する
- `7.6 → 7.7` のような小差で大きく見えすぎないことを確認する
- 未測定バーが同一幅になることを確認する
- 改善方向が正しく出ることを確認する

### Step 6: 進行状況イベントの設計

次の4種類を扱える状態にする。

- 工程開始
- 工程完了
- 工程スキップ
- 工程中の内部進捗

### Step 7: 補正処理から工程イベントを出す

- 既存ログは残す
- 進行表示はログ文に依存しない
- 各工程の開始・完了・スキップを明示する

### Step 8: マスタリング処理から工程イベントを出す

- 補正側と同じ形式にする
- ノイズ戻りなど長い工程は内部進捗も可能な範囲で出す

### Step 9: 進行状況UIを整理

- 未開始、実行中、完了、スキップを見分けられるようにする
- スキップは「失敗」ではなく「不要だった」と分かる見た目にする
- 長い処理中に止まって見えないようにする

### Step 10: 進行状況テストを追加

- 工程開始で現在工程になる
- 工程完了でチェックになる
- スキップで灰色表示になる
- ログ文が変わっても進行状態が壊れない

### Step 11: `LoudnessMeasurementService` を追加

次の3つを1つの測定サービスにまとめる。

- LUFS
- True Peak
- LRA

### Step 12: LUFS の測定基準を統一

- K-weighting
- 400msブロック
- 100ms hop
- -70 LUFS 絶対ゲート
- 相対 -10 LU ゲート
- 左右チャンネル合算

### Step 13: True Peak の測定方法を固定

- 4倍以上のオーバーサンプリング
- サンプル間ピークを見る
- 近似の範囲を明確にする
- 表示名を実装内容に合わせる

### Step 14: LRA の測定方法を統一

- LUFSと同じ短期ラウドネスを使う
- 静かすぎる区間を除外する
- 下位10%〜上位95%の差を使う

### Step 15: 既存サービスを接続し直す

- `AudioComparisonService` は共通測定結果を使う
- `MasteringAnalysisService` は必要な測定値を共通サービスから使う
- 表、グラフ、品質チェックの値を統一する

### Step 16: 音量系テストを追加

- 同じ波形なら同じLUFSになる
- 無音混じりでゲートが効く
- ステレオ合算が効く
- True Peak がサンプルピークだけに依存しない
- 平坦な音源でLRAが小さくなる
- 変化の大きい音源でLRAが大きくなる

### Step 17: 全体テスト

次を実行する。

- 関連テスト
- 既存テスト
- `swift test`

| 領域 | 完了条件 |
|---|---|
| ノイズ数値 | 入力・補正後・最終版の実測値がそのまま表示される |
| ノイズバー | 数値差以上に極端な見た目にならない |
| 未測定バー | 全行で同じ長さ、同じ薄さになる |
| 説明 | 各ノイズの改善方向が分かる |
| 判定 | バーではなく実測値と差分から出る |
| 進行状況 | 工程ごとに開始・完了・スキップが見える |
| 長い処理 | 止まって見えない |
| LUFS | 1つの測定処理から使われる |
| True Peak | 測定方法が固定される |
| LRA | LUFSと同じ基準で出る |
| UI | 目安と正式測定が混ざらない |
| テスト | 追加テストと `swift test` が通る |

## 12. この計画でやらないこと

| やらないこと | 理由 |
|---|---|
| ノイズ実測値を見た目に合わせて変える | 実測値の信頼性が落ちる |
| 音声処理そのものを同時に大きく変える | 表示修正と音質修正が混ざる |
| バーを完全な測定器として扱う | 正確な判断は数値と差分で行うべき |
| ログ文字列だけで進行状況を制御する | 文言変更に弱い |
| LUFS / True Peak / LRA を複数箇所で別々に計算する | 表示差の原因になる |

## 最終方針

実測値は変えず、表示は誤解なく、進行状況は工程単位で、音量系指標は同じ測定基準で出す。

この方針で進めれば、簡易的な見た目調整ではなく、画面として信用できる実用レベルの修正になる。
